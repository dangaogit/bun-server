# Bun Server Framework - v2.0.0 Roadmap

> Target version: v2.0.0\
> Focus: AI Modules — 9 official AI-related modules, general AI platform capabilities, packages/web promotional portal\
> **Status**: 🚧 In Progress

---

## Overview

v2.0.0 is a major release introducing first-class AI support to Bun Server Framework. It adds 9 new official modules that together enable building production-grade AI applications — from simple LLM chat to full RAG pipelines, Tool Calling agents, and MCP protocol servers.

The design is based on a practical capability analysis for modern AI application platforms. All infrastructure-level capabilities are provided as official modules; business-level orchestration (ReAct agents, AI workflows) is demonstrated in the `ai-platform-mvp` example.

## New Official Modules

| Module | Token | Purpose |
|--------|-------|---------|
| `AiModule` | `AI_SERVICE_TOKEN` | LLM unified access (OpenAI / Claude / Gemini / Ollama) + Tool Calling |
| `ConversationModule` | `CONVERSATION_SERVICE_TOKEN` | Multi-turn conversation memory |
| `PromptModule` | `PROMPT_SERVICE_TOKEN` | Prompt template management + versioning |
| `EmbeddingModule` | `EMBEDDING_SERVICE_TOKEN` | Text embedding generation |
| `VectorStoreModule` | `VECTOR_STORE_TOKEN` | Vector similarity search |
| `RagModule` | `RAG_SERVICE_TOKEN` | Full RAG pipeline (ingest → retrieve → generate) |
| `McpModule` | `MCP_SERVER_TOKEN` | MCP protocol server (JSON-RPC 2.0, SSE/stdio) |
| `AiGuardModule` | `AI_GUARD_SERVICE_TOKEN` | AI content safety (PII, moderation, prompt injection) |

> Note: `PromptModule` does not have a dedicated token entry; it uses `PROMPT_SERVICE_TOKEN` exported from `src/prompt/types.ts`.

## Key Design Decisions

### Zero External Dependencies

All AI providers communicate via Bun's native `fetch()` and direct HTTP APIs. No third-party AI SDK (openai, @anthropic-ai/sdk, etc.) is added as a dependency. This keeps the framework lightweight and compatible with any OpenAI-compatible endpoint.

### Abstract-First Design

Every AI module follows the same abstract-first pattern used in `src/microservice/`:
1. Define abstract interface (`LlmProvider`, `ConversationStore`, etc.)
2. Provide built-in implementations (Memory, File)
3. Allow users to plug in external stores/providers

### ConversationModule Summarizer Decoupling

`ConversationService.summarize()` does not directly depend on `AiModule` to avoid circular dependencies. Users inject a `summarizer` callback via `forRoot()`:

```typescript
ConversationModule.forRoot({
  store: MemoryConversationStore,
  summarizer: async (history) => await aiService.summarize(history),
});
```

### Streaming Responses

`AiService.stream()` returns `ReadableStream<Uint8Array>` (SSE-encoded). Controllers can return it directly as a `Response`:

```typescript
@GET('/stream')
public stream(): Response {
  const stream = this.aiService.stream({ messages });
  return new Response(stream, {
    headers: { 'Content-Type': 'text/event-stream' },
  });
}
```

## Migration from v1.x

v2.0.0 is **backward compatible** with v1.x. All existing modules (`ConfigModule`, `SecurityModule`, `DatabaseModule`, etc.) continue to work without changes.

New AI modules are purely additive. No breaking changes to existing public APIs.

See `docs/migration.md` for full v1.x → v2.0 migration guide.
